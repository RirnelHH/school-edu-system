## Context

技工院校教务管理系统，需要支撑：
- 1000-5000学生 + 100-500教师
- 多角色（学生/教师/班主任/教研组长/主任/管理员）
- 核心流程：培养计划→排课→成绩→请假→课时统计

技术约束：
- 本地化部署（学校有服务器）
- 简化运维（单一Docker镜像）
- 微信小程序（移动端优先）

## Goals / Non-Goals

**Goals:**
- 统一教务平台数字化
- 审批流程自动化
- 课时计算精确
- 移动端支持（小程序）
- 部署简单（Docker）

**Non-Goals:**
- 不做考试管理/评教/毕业管理
- 不做多语言
- 不做App（先做小程序）

## Decisions

### 1. 后端架构：NestJS单体应用
**选择理由：**
- TypeScript统一语言
- 模块化易维护
- 生态成熟（Prisma/TypeORM）
- 对比微服务：简单、运维成本低

### 2. 前端：Vue3管理后台 + Taro小程序
**选择理由：**
- Vue3生态好，Element Plus配套
- Taro跨端方案成熟，支持微信小程序
- 小程序即移动端入口

### 3. 数据库：MySQL 8.0
**选择理由：**
- 关系型数据为主
- 成熟稳定
- 本地部署方便

### 4. ORM：Prisma
**选择理由：**
- 现代化、类型安全
- 迁移方便
- Schema即文档

### 5. 部署：Docker单一镜像
**选择理由：**
- 简化部署
- 一键启动
- 对比K8s：运维简单

### 6. 认证：JWT + 微信OAuth
**选择理由：**
- JWT无状态，适合API
- 微信OAuth小程序标配

## Risks / Trade-offs

**[Risk] 微信OAuth依赖**
- 微信接口变动可能影响登录
-  Mitigation：准备手机号+密码备用登录

**[Risk] 排课算法复杂度**
- 多人多约束排课可能计算慢
-  Mitigation：先做手动调整，后续优化算法

**[Risk] 课时系数计算规则**
- 规则可能经常调整
-  Mitigation：系数可配置化
